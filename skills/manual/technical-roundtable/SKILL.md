## 适用场景

当需要组织技术专家对EvoAgent的设计方案进行评审时使用此技能。

## 典型使用场景

1. **新增功能设计评审** - 如技能进化系统
2. **架构变更评审** - 如并发控制、通信机制
3. **生产就绪评估** - 如可观测性、稳定性机制
4. **问题整改评审** - 针对专家反馈的问题进行整改后的确认

## 会议组织流程

### 1. 确定评审范围

```bash
# 确定需要评审的文档模块
evoagent docs:ls                    # 列出所有文档
evoagent docs:cat design.md         # 查看当前设计版本
```

评审范围选项：
- `docs/design.md` - 整体架构设计
- `docs/implementation-plan.md` - 实施计划
- 特定模块（如 `进化系统`、`记忆系统`）

### 2. 选择专家

根据评审内容选择合适的专家：

| 评审内容 | 推荐专家 |
|---------|---------|
| 整体架构、并发控制 | ClawdBot作者、Mauns核心开发 |
| CLI、用户体验 | OhMyOpenCode作者 |
| 记忆系统、向量搜索 | OpenMemory、PageIndex核心开发、Chroma.js专家 |
| 遗忘曲线、记忆巩固 | 神经科学家 |
| 生产可观测性 | SRE专家 |
| 数据库、存储 | 数据库专家 |

### 3. 准备评审材料

```markdown
## 评审邀请模板

### 评审对象
- 文档: docs/design.md
- 版本: v2.1
- 章节: [具体章节名称]
- 状态: 设计阶段 / 生产就绪

### 评审问题
1. [具体问题1]
2. [具体问题2]
3. [具体问题3]

### 背景信息
- 相关链接: [设计文档链接]
- 上轮反馈: [如果有]
- 变更摘要: [简要说明本次变更内容]

### 预期输出
- 问题清单 (优先级 P0/P1/P2)
- 改进建议
- 是否通过评审
```

### 4. 会议主持

#### 开场

```
各位专家好，感谢参与EvoAgent v{version} {主题}评审会。

今天会议的目标是：
1. 评审 {具体设计方案}
2. 识别潜在风险和问题
3. 收集改进建议
4. 确定是否通过评审或需要整改

会议预计时长：{时长}

让我们开始。
```

#### 会议中

| 任务 | 说明 |
|------|------|
| **记录问题** | 按P0/P1/P2优先级记录 |
| **确认理解** | 复述专家观点确认理解正确 |
| **追问细节** | 对重要观点进行深入探讨 |
| **总结归纳** | 定期总结当前共识 |

#### 收尾

```
感谢各位专家的宝贵反馈。

### 评审结果
- 共收到问题：{数量} 个
- P0问题：{数量} 个
- P1问题：{数量} 个
- P2问题：{数量} 个

### 下一步
- {整改计划或下一步行动}

预计 {时间} 完成整改，随后进行下一轮评审。
```

## 相关技能

- `design-review` - 设计文档评审技能
- `code-review` - 代码审查技能
- `expert-coordination` - 专家协调技能

## 注意事项

1. **会议前**：确保所有专家能够访问最新的设计文档
2. **会议中**：保持中立，不偏袒任何一方
3. **会议后**：及时整理评审记录，分享给所有参与者
4. **间隔**：两轮评审之间至少预留1天整改时间

## 反模式（不适用场景）

- 不需要评审的内部讨论（使用普通会议即可）
- 日常代码审查（使用 code-review 技能）
- 紧急故障复盘（使用 incident-review 技能）
